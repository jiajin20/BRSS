# 2026-05-25 工作记录

## BRSS 项目 RSS 服务 Bug 检查

对 `bili_rss/app.py` 进行了完整的静态代码审查，发现以下问题：

### 高危 Bug
1. **线程安全 - db.json 并发写入冲突**：`download_audio` 和 `download_up_videos` 运行在后台线程，同时调用 `load_db()` + `save_db(db)`，无任何锁机制，并发下载时存在数据丢失风险。
2. **generate_rss() 中 base_url 硬编码**（第410行）：`base_url = 'http://服务器IP'`，本地运行时 RSS enclosure URL 指向错误地址，音频无法播放。

### 中级 Bug
3. **任务保存后再重复 load_db()**（第270行）：`download_audio` 函数末尾先 `load_db()` 取到旧数据，再写回，有概率覆盖其他线程写入的 bv_ids。
4. **BV ID 提取逻辑双重冗余**（第183行）：`extract_bv_id` 先 regex 匹配再额外判断 `startswith('BV')`, 第二个分支与 regex 有重叠但要求长度>=10，可能漏掉合法短 BV 号。
5. **`api_delete_audio` 未从 db.tasks 中移除 bv_id**：删除音频文件 + 从 collections 移除后，tasks 的 bv_ids 仍保留该 BV，导致页面统计数不准确。
6. **batch-delete 同样未清理 tasks 中的 bv_ids**（同上）。

### 低级 / 体验问题
7. **`read_cookie_str_from_file` 裸 except 吞异常**（第173行）：解析 cookie 文件出错时静默返回空字符串，无日志，难以调试。
8. **`resolve_cookie_file` 逻辑**（第150行）：若 IP 专属 cookie 文件已存在，即使用户本次传入了新 cookie，也不会更新。
9. **`download_up_videos` 解析 succeeded_in_batch 脆弱**（第385行）：用字符串 replace+split 解析进度消息，格式变化会静默出错。
10. **`generate_rss()` 无 try-except 包裹**：RSS_DIR 不存在时直接抛出 FileNotFoundError，导致 API 返回 500 而没有友好提示。
11. **`run_local.py` 的 monkey-patch 在模块导入后才执行**：`BASE_DIR` 等被 patch 了，但 `generate_rss` 函数内硬编码的 `base_url = 'http://服务器IP'` 不受影响，本地 RSS 地址仍然错误。

## 高危 Bug 修复（已完成）

### Bug 1 修复 - 加 db_lock
- 在模块全局区添加 `db_lock = threading.Lock()`
- `download_audio()` 末尾 load+save 用 `with db_lock:` 包裹（L277）
- `download_up_videos()` 中 load+save 用 `with db_lock:` 包裹（L373）

### Bug 2 修复 - BASE_URL 动态获取
- 新增 `get_base_url()` 函数：有请求上下文时读 `Host` 头拼 URL，无上下文时用 `_last_known_base_url` 或环境变量 `BILI_RSS_BASE_URL` 兜底
- `generate_rss()` 改用 `get_base_url()` 动态获取地址
- `BASE_URL` 支持环境变量 `BILI_RSS_BASE_URL` 配置
- 顺带修复 Bug 9：`generate_rss()` 开头加 `RSS_DIR.mkdir(parents=True, exist_ok=True)`

### 密钥管理重构（菜单 9/10）
- **原逻辑错误**：查看/修改的是一台机器本地的 SSH 密钥路径（与项目无关）
- **修正后**：管理的是服务器 `app.py` 中硬编码的 `DELETE_SECRET_KEY`（删除音频文件时的验证密码）
- `do_view_key(ssh, cfg)`: SSH 远程执行 Python 读取并显示当前密钥
- `do_modify_key(ssh, cfg)`: 
  1. 先显示当前密钥（首8+末4可见，中间*号遮罩）
  2. 输入新密钥（至少8位）
  3. 用远程 Python re.sub 安全替换（避免 sed 特殊字符问题）
  4. systemctl restart 重启服务使密钥生效
- `MENU_DISPLAY` 中 9/10 的 `needs_ssh` 从 `False` 改为 `True`
- exe 已重新打包到 `deploy/dist/BiliRSS-Deploy.exe`（13.9 MB）

## 修改密钥 shell 特殊字符转义修复
- **问题**：密钥 `Ah+$ZnbC-#I.<rzzdtXX$,l?*0SYXfu{7dtgEjR+<p9]hi$n,aSxhH5A]f]muGNO` 包含 `$#.'"` 等特殊字符，`python3 -c "..."` 方式直接嵌入时 shell 转义崩溃，报 `unexpected EOF`
- **修复**：
  1. 本地 `base64.b64encode()` 编码密钥
  2. SFTP 写入临时 `.py` 脚本到服务器（`{remote_base}/.deploy_update_key.py`）
  3. 远程执行 `python3 tmp_script`，脚本内 base64 解码 + re.sub 替换
  4. finally 清理临时脚本
  5. `do_modify_key` 签名改为 `(ssh, sftp, cfg)` 以获取 sftp 连接
- exe 已重新打包：`deploy/dist/BiliRSS-Deploy.exe`
